# Abnormal Security Cases

## Overview

In addition to AI-detected threats, Abnormal Security manages security cases — structured investigations that may span multiple related threats or user reports. The platform also provides an Abuse Mailbox feature, where users report suspicious emails directly, and Abnormal analyzes them for threats. This skill covers listing and managing security cases and processing user-reported phishing emails (abuse reports).

## Key Concepts

### Security Cases

Security cases in Abnormal aggregate related threat activity into a single investigation context. Cases may be:
- Auto-generated by Abnormal AI when multiple related threats are detected
- Created by security analysts who manually group related threats

Cases have a severity and status that tracks the investigation lifecycle.

### Abuse Reports (User-Reported Phishing)

The Abnormal Abuse Mailbox allows users to report suspicious emails by forwarding them to a designated address. Abnormal ingests these reports and runs its AI analysis against the reported message. Each abuse report receives a verdict:
- **Malicious** — Abnormal confirmed the reported email is an attack
- **Suspicious** — The email exhibits some attack indicators but is not conclusive
- **Safe** — The email is legitimate (false positive report from the user)
- **Unknown** — Insufficient information for a verdict

Processing abuse reports promptly reduces dwell time for threats that bypassed initial detection.

### Case Status

| Status | Description |
|--------|-------------|
| `OPEN` | Active investigation requiring action |
| `IN_PROGRESS` | Under active investigation |
| `CLOSED` | Resolved and closed |

## API Patterns

### List Security Cases

```
abnormal_list_cases
```

Parameters:
- `filter` — Filter expression (e.g. `status eq OPEN`, `severity eq HIGH`)
- `pageNumber` — Page number for pagination

**Example — List open high-severity cases:**

```json
{
  "filter": "status eq OPEN and severity eq HIGH"
}
```

**Example response:**

### Get Abuse Reports

```
abnormal_get_abuse_reports
```

Returns user-reported phishing emails submitted to the Abnormal Abuse Mailbox.

Parameters:
- `filter` — Filter expression (e.g. `reportedTime gte 2026-03-01`, `verdict eq MALICIOUS`)
- `pageNumber` — Page number for pagination

**Example — Get today's abuse reports:**

```json
{
  "filter": "reportedTime gte 2026-03-02T00:00:00Z"
}
```

**Example response:**


## Common Workflows

### Daily Abuse Report Review

1. Call `abnormal_get_abuse_reports` with `filter` for the past 24 hours
2. Categorize by verdict:
   - **MALICIOUS** — Confirmed threat; verify `remediationStatus`
   - **SUSPICIOUS** — Requires manual review; investigate further
   - **SAFE** — False positive; send user reassurance response
   - **UNKNOWN** — Insufficient data; review manually in the Abnormal portal
3. For MALICIOUS reports with `remediationStatus=NOT_REMEDIATED`:
   - Cross-reference with `linkedThreatId` and call `abnormal_manage_remediation`
4. For SUSPICIOUS reports:
   - Review the `attackDescription` and call `abnormal_get_threat` on `linkedThreatId` if present
   - Investigate the original sender domain and linked URLs
5. Compile a summary of confirmed threats and false positives for client reporting

### Respond to a User Phishing Report

1. Receive user report (email to abuse mailbox or helpdesk ticket)
2. Call `abnormal_get_abuse_reports` with `filter` matching the reporter's email and recent time
3. Find the matching report and check `verdict`
4. If MALICIOUS and `remediationStatus=REMEDIATED`: inform user the threat has been handled
5. If MALICIOUS and `remediationStatus=NOT_REMEDIATED`: trigger `abnormal_manage_remediation`
6. If SAFE: reassure user and explain why the email is legitimate
7. If SUSPICIOUS: escalate for manual investigation

### Review Open Security Cases

1. Call `abnormal_list_cases` with `filter=status eq OPEN`
2. Sort by `severity` (HIGH first)
3. For each case, review `threatIds` and `affectedUsers`
4. Call `abnormal_get_threat` for each linked threat ID to understand the full picture
5. Update the case status or escalate based on findings
6. For cases with multiple affected users, notify all impacted clients simultaneously

### Track Campaign Scope

1. Identify a phishing campaign from `abnormal_list_threats` or abuse reports
2. Call `abnormal_list_cases` to find a case that groups related threats
3. If no case exists, identify all threats with the same sender domain or URL pattern
4. Aggregate the full list of affected users across all related threats
5. Report campaign scope and remediation status to the client

## Error Handling

### Abuse Report Not Found for a User-Reported Email

**Cause:** The Abnormal Abuse Mailbox may not be configured for the tenant, or the user reported via a different channel.
**Solution:** Verify the Abuse Mailbox is enabled in the Abnormal Security Portal. If users are reporting via helpdesk instead, cross-reference with `abnormal_list_threats` using sender/recipient.

### SUSPICIOUS Verdict with No Linked Threat

**Cause:** Abnormal flagged the email as suspicious but did not create a full threat case.
**Solution:** Review the `attackDescription` for indicators and investigate manually. If indicators are strong, check the sender domain reputation and linked URLs using external threat intelligence tools.

### Case Aggregates Unrelated Threats

**Cause:** Abnormal's case grouping algorithm may occasionally group coincidentally related threats.
**Solution:** Review each `linkedThreatId` independently — do not assume all threats in a case are part of the same campaign.

## Best Practices

- Process abuse reports within 2 hours — user-reported phishing can indicate active campaigns that bypassed initial detection
- Always check `verdict` before acting — SAFE verdicts should not trigger remediation
- For MALICIOUS abuse reports, check `linkedThreatId` — if linked, the threat may already be remediated
- Communicate SAFE verdicts to reporting users to reduce future false positive reports and improve user security awareness
- Track the ratio of MALICIOUS vs. SAFE abuse reports per tenant — a high SAFE ratio indicates users need phishing awareness training
- High-severity open cases with multiple affected users are BEC campaigns — notify client leadership immediately
